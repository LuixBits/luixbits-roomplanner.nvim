1. INTRODUCTION                                      *roomplan-introduction*

RoomPlan stores exact millimetre geometry in structured JSON or a marked
Neorg block and renders an interactive terminal view. Display rounding,
zooming, aspect calibration, and view rotation never alter saved geometry.

Version 1 supports one floor, rectangular rooms, rectangular furniture,
shared-wall doors, and furniture rotations in 90-degree steps. It is a space
planner, not CAD/BIM software or a building-code checker.

Requirements:

- Neovim 0.10 or newer
- no mandatory runtime dependencies
- Neorg only when embedding plans in .norg files

The complete linked handbook is in docs/README.md in the repository.

2. INSTALLATION                                             *roomplan-install*

The repository must be on 'runtimepath' before Lua calls require("roomplan").
The plugin loader registers commands automatically; setup() is optional when
the defaults are sufficient.

lazy.nvim: >lua
  {
    "LuixBits/luixbits-roomplanner.nvim",
    lazy = false,
    main = "roomplan",
    opts = {},
  }
<

Neovim 0.12 vim.pack: >lua
  vim.pack.add({
    { src = "https://github.com/LuixBits/luixbits-roomplanner.nvim" },
  })
  require("roomplan").setup({})
<

For Nix flakes, nvf, rocks.nvim, native packages, and local checkout examples,
see docs/getting-started/installation.md.

3. FIRST PLAN                                            *roomplan-quickstart*

Create a standalone plan without overwriting an existing file: >vim
  :RoomPlanInit ~/plans/flat.roomplan.json
<

Press `a` to add the first room. In the form, move with `j`/`k`, edit with
<CR>, and apply with <C-s>. Back on the canvas, use `h`/`j`/`k`/`l`, press
`e` for exact edits, and `s` to save.

Open the plan later with: >vim
  :RoomPlanOpen ~/plans/flat.roomplan.json
<

Inside a .norg buffer, |:RoomPlanInit| initializes a marked embedded block and
|:RoomPlanOpen| discovers it. A standalone path may always be supplied
explicitly.

5. WORKSPACE                                             *roomplan-workspace*

The workspace has a canvas, a left Navigator/Issues pane, an optional right
Details pane, and a one-line contextual footer. The Navigator starts visible;
Details starts hidden. Both retain their selection and can be toggled to give
the canvas more room.

Wide terminals use side panes. Medium layouts retain only the panes that fit.
Compact layouts open bordered drawers. Manually closing a pane keeps it closed
until explicitly toggled again.

Navigator shows rooms with nested doors and furniture. Issues filters and
selects diagnostics. Details presents compact, collapsible sections whose
contents change with the selection. `?` always exposes the full action set,
including disabled actions and their reasons.

Structured forms keep their complete draft visible. A form is revision-bound:
if the plan changes elsewhere, the stale form cannot apply over newer data.

6. KEYMAPS                                                 *roomplan-keymaps*

Workspace:

  1 / o          focus or toggle Navigator
  2              focus canvas
  3 / i          focus or toggle Details
  !              focus or toggle Issues
  <Tab>          next pane
  <S-Tab>        previous pane
  <CR>           select a Navigator/Issue row; toggle Details section
  /              filter Navigator or Issues
  h / l          collapse / expand a room or Details section
  ?              complete action palette or contextual help
  q              return to canvas, close a drawer, or hide workspace
  <Esc>          cancel the current interaction, then return to canvas

Canvas and editing:

  h j k l        move cursor/object, or pan in PAN mode
  H J K L        coarse movement
  <C-h/j/k/l>    fine movement
  <CR>           select object under cursor
  m / p          MOVE / PAN mode
  a              Add palette
  D / F          add door / furniture
  A              align room
  e              exact edit
  d / y / r      delete / duplicate / rotate furniture
  v / ]e / [e    validate / next issue / previous issue
  u / <C-r>      undo / redo
  z+ / z- / f    zoom in / zoom out / fit
  zh zj zk zl    pan without entering PAN mode
  ]r / [r / g0   rotate view clockwise / counterclockwise / north-up
  gs / g!        toggle snapping / bypass the next snap
  s / S          save / save as

Forms:

  j k / Tab      next / previous field
  <CR> / e       edit the active field
  h l            cycle a choice
  <Space>        toggle or advance a choice
  <C-s>          apply atomically
  R              reset the form draft
  q / <Esc>      cancel

Set keymaps.enabled=false to install no RoomPlan mappings. Override a semantic
mapping name or its default left-hand side in keymaps.mappings. Use false to
disable one mapping. Displayed hints use the same resolver as installed maps.

7. CONFIGURATION                                           *roomplan-config*

Call require("roomplan").setup({...}). Unknown options and invalid values are
errors; options are not silently ignored. The important defaults are: >lua
  {
    plan_defaults = {
      metadata = { name = nil, notes = "" },
      settings = {
        grid_mm = 100,
        fine_step_mm = 10,
        normal_step_mm = 100,
        coarse_step_mm = 500,
        default_door_width_mm = 900,
      },
    },
    furniture = { include_builtins = true, definitions = {}, files = {} },
    canvas = {
      open = "tab",
      unicode = "auto",
      mm_per_column = 100,
      cell_aspect = 2.0,
      zoom_factor = 1.25,
      min_mm_per_column = 1,
      max_mm_per_column = 100000,
      fit_margin_cells = 2,
      header_lines = 1,
      pan_step_cells = 5,
      pan_coarse_step_cells = 20,
      show_grid = false,
      show_dimensions = true,
      show_compass = true,
    },
    snapping = {
      enabled = true,
      tolerance_cells = 1.5,
      max_distance_mm = 250,
      priority = { "door", "room_edge", "room_center", "furniture", "grid" },
    },
    ui = {
      confirm_delete = true,
      notify_level = "info",
      workspace = {
        layout = "auto",
        left_width = 26,
        right_width = 30,
        navigator_visible = true,
        details_visible = false,
        wide_min_columns = 120,
        compact_max_columns = 89,
        compact_min_rows = 22,
        min_canvas_width = 55,
        min_canvas_height = 10,
        footer_height = 1,
        cycle_tabs = true,
        ascii = false,
        border = "rounded",
      },
    },
    autosave = { enabled = false, debounce_ms = 1000, norg = false },
    keymaps = { enabled = true, mappings = {} },
    glyphs = nil,
  }
<

Resource ceilings live under limits. See docs/configuration/settings.md for
the complete list and docs/configuration/keymaps.md for semantic mapping names.

Highlight groups start with RoomPlan and use default links, so colorschemes or
user configuration can override them normally. See docs/display/appearance.md.

8. FURNITURE CATALOGUES                                 *roomplan-furniture*

RoomPlan includes generic built-in templates. Add process-wide defaults inline
or load one or more dependency-free JSON files. Set
furniture.include_builtins=false to make imported and inline definitions the
complete visible catalogue. Built-in IDs remain resolvable for existing plans.
Project-local templates saved inside a plan take precedence over imported
templates with the same custom ID; built-in IDs remain reserved.

Inline example: >lua
  require("roomplan").setup({
    furniture = {
      definitions = {
        {
          id = "custom:standing-desk",
          name = "Standing desk",
          category = "work",
          shape = "rectangle",
          default_size_mm = { 1600, 800, 1250 },
        },
      },
      files = { vim.fn.stdpath("config") .. "/roomplan-furniture.json" },
    },
  })
<

File format: >json
  {
    "version": 1,
    "furniture": [
      {
        "id": "custom:reading-chair",
        "name": "Reading chair",
        "category": "seating",
        "shape": "rectangle",
        "default_size_mm": [850, 900, 1000]
      }
    ]
  }
<

Catalogue setup is atomic: an invalid definition or file leaves the previous
effective catalogue unchanged. Files are limited to 1 MiB.

9. ASPECT AND VIEW ROTATION                               *roomplan-display*

Terminal cells are usually taller than they are wide. canvas.cell_aspect is
cell height divided by cell width; the default is 2.0. If a square room looks
vertically stretched, use :RoomPlanAspect with a measured ratio. This updates
the current Neovim process and refits every live canvas; no rebuild is needed.

View rotation is also transient. ]r and [r rotate by 90 degrees, g0 restores
north-up, and the compass shows where world north points. Movement and panning
remain screen-relative. Saved room, door, and furniture coordinates never
rotate.

10. STORAGE AND SAFETY                                    *roomplan-storage*

Standalone files conventionally end in .roomplan.json. Neorg plans use a
marked ranged tag. RoomPlan preserves unknown extension fields and refuses
ambiguous or unsafe source operations.

One writable session owns a source at a time. Other plans can remain open.
Writes use revision checks and staged/atomic replacement where applicable.
Source changes produce a conflict instead of silently overwriting either side.
Dirty sessions are protected on close, reload, :q, and process exit.

Structural schema failures cannot be saved. Layout-invalid drafts may be
reviewed in Issues, repaired, or explicitly saved with !. That escape hatch is
not a claim that the layout is safe or physically constructible.

The current schema is schema/roomplan.schema.json. Geometry uses integer
millimetres; derived door geometry may use doubled-millimetre arithmetic so
half-millimetre endpoints stay exact.

11. LUA API                                                   *roomplan-api*

Public entry points: >lua
  local roomplan = require("roomplan")

  roomplan.setup(opts)
  roomplan.open(opts, callback)
  roomplan.init(opts, callback)
  roomplan.save(opts, callback)
  roomplan.save_as(path, opts, callback)
  roomplan.reload(opts, callback)
  roomplan.set_aspect(ratio, opts, callback)
  roomplan.rotate_view(direction)
  roomplan.hide(opts)
  roomplan.close(opts, callback)
  roomplan.validate(opts)
  roomplan.sessions()
<

roomplan.aspect is retained as a compatibility alias for set_aspect; new
configuration should use the explicit canonical name.

Callbacks receive (result, error). With no callback, noninteractive operations
return those values directly. Errors are structured tables with at least code
and message where possible.

For low-level integrations, require("roomplan.api").dispatch(session_id,
action) sends a model action through the normal controller/history boundary;
session(session_id) returns the live session.

12. HEALTH AND TROUBLESHOOTING                            *roomplan-troubles*

Run: >vim
  :checkhealth roomplan
<

It checks Neovim/runtime compatibility, encoding, display capabilities,
configuration and mappings, optional Neorg, workspace sizing, live sessions,
and source writability.

If require("roomplan") says the module is missing, the package is not on
'runtimepath' at the time configuration executes. Fix the package-manager or
Nix/nvf wiring; do not add a separate Lua package. If :RoomPlan is missing,
the plugin loader was not sourced for the same reason. Restart Neovim after
changing the installed package/generation.

If the canvas looks stretched, run :RoomPlanAspect; this never requires a Nix
or system rebuild. If a plan is not visible, press `f`. If an action is not in
the compact footer, press `?` to see the complete contextual palette.

For detailed diagnosis see docs/reference/troubleshooting.md.
